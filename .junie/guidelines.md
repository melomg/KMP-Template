# KMP-Template: Project-specific Development Guidelines

These guidelines capture project-specific knowledge to help advanced contributors work efficiently on Now in Android within this repository.

## Architecture

This project is a modern Android application that follows the official architecture guidance from Google. It is a reactive, single-activity app that uses the following:

-   **UI:** Built entirely with Jetpack Compose, including Material 3 components and adaptive layouts for different screen sizes.
-   **State Management:** Unidirectional Data Flow (UDF) is implemented using Kotlin Coroutines and `Flow`s. `ViewModel`s act as state holders, exposing UI state as streams of data.
-   **Dependency Injection:** Hilt is used for dependency injection throughout the app, simplifying the management of dependencies and improving testability.
-   **Navigation:** Navigation is handled by Jetpack Navigation 2 for Compose, allowing for a declarative and type-safe way to navigate between screens.
-   **Data:** The data layer is implemented using the repository pattern.
    -   **Local Data:** Room and DataStore are used for local data persistence.
    -   **Remote Data:** Retrofit and OkHttp are used for fetching data from the network.
-   **Background Processing:** WorkManager is used for deferrable background tasks.

## Build and Configuration

### Toolchains and prerequisites
- JDK: 17 (project compiles and runs with Java 17). The composeApp Android target sets JVM target 17 and Android compileOptions to 17.
- Gradle Wrapper: use the provided ./gradlew. Configuration cache and parallel build are enabled in gradle.properties; prefer using the wrapper to preserve compatibility.
- Android: Android SDK matching versions in gradle/libs.versions.toml; debug builds do not require signing config. Release signing reads from local.properties; see 1.3.
- iOS: Xcode 15+ recommended. Kotlin Multiplatform + Compose Multiplatform with frameworks built by Gradle and consumed by iosApp.
- Desktop: JVM target via Compose Desktop. No native toolchain required beyond JDK.
- Web (Wasm): Kotlin/Wasm enabled (org.jetbrains.compose.experimental.wasm.enabled=true). Dev server runs via :composeApp:wasmJsBrowserDevelopmentRun.

### Repository modules
- composeApp: Compose Multiplatform app code with targets android, ios, desktop, wasmJs. Uses BuildKonfig to expose build constants from application.properties.
- server: Ktor server (Netty). Easiest place to run fast unit tests locally.
- shared:model and shared:network: KMP shared code, used by composeApp.

### Configuration inputs
- application.properties (root) controls app identifiers, names, and versions. BuildKonfig wires these into com.melih.kmptemplate.BuildKonfig fields (e.g., APP_NAME, EFFECTIVE_BUILD_TYPE, IS_DEBUGGABLE, etc.). Effective build type is derived from Gradle tasks or environment variables:
  - From Gradle tasks: Debug/Staging/Release in the task requests.
  - iOS: CONFIGURATION environment variable.
  - Fallback: EFFECTIVE_BUILD_TYPE env var (debug|staging|release) or release if unset.
- Google/Firebase: The project expects composeApp/google-services.json to exist (already present). This is necessary for Android build. The Google Services Gradle plugin is applied in composeApp.
- Signing (Android):
  - debug/staging: uses debug keystore; no extra setup required.
  - release: requires local.properties with keys: storePassword, keyAlias, keyPassword. The keystore file path is tools/release.jks. Example local.properties entries:
    storePassword=... 
    keyAlias=... 
    keyPassword=...

### Build commands (verified where feasible)
- Server (Ktor):
  - Run tests: ./gradlew :server:test
  - Run server app: ./gradlew :server:run (serves on 0.0.0.0:8080; GET / returns "Ktor").
- Web (Wasm) dev:
  - Run dev server: ./gradlew :composeApp:wasmJsBrowserDevelopmentRun
- Desktop:
  - Run desktop app: ./gradlew :composeApp:run
- Android:
  - Assemble debug APK/AAB: ./gradlew :composeApp:assembleDebug
  - Install on device: ./gradlew :composeApp:installDebug (Android SDK/emulator required)
- iOS:
  - Open iosApp/iosApp.xcodeproj in Xcode and select iosDebug or iosStaging/iosRelease scheme. Compose UI is embedded via shared Compose framework generated by Gradle. Alternatively, build frameworks from Gradle first (Xcode does this via Kotlin Gradle plugin integration when running).

Notes:
- Detekt static analysis is applied to all subprojects via root build.gradle.kts. It uses tools/detekt/detekt-config.yml. ignoredBuildTypes = ["staging", "release"] reduces noise on non-debug variants.
- Gradle optimizations (config cache, build cache, parallel) are enabled; if you add custom Gradle tasks/plugins ensure they are configuration-cache compatible.

## Testing: configuration, running, and extending

### Test frameworks in use
- server module: kotlin.test with JUnit engine and Ktor test host (testImplementation(libs.ktor.serverTestHost), testImplementation(libs.jetbrains.kotlin.testJunit)). Tests run on JVM quickly and are the recommended place for fast iteration.
- composeApp: commonTest uses kotlin.test. Running these across multiplatform targets can require additional setup; prefer server tests for quick validation unless you specifically need platform code coverage.

### How to run tests (verified)
- All server tests: ./gradlew :server:test
- Single test class (server): ./gradlew :server:test --tests "com.melih.kmptemplate.ApplicationTest"
- Single test method (server): ./gradlew :server:test --tests "com.melih.kmptemplate.ApplicationTest.testRoot"

### Adding a new test (server)
- Create a test file under server/src/test/kotlin/com/melih/kmptemplate.
- Example (we verified this flow locally):
  - File: SanityTest.kt
    class SanityTest {
        @Test fun sanity() { assertTrue(2 + 2 == 4) }
    }
  - Run: ./gradlew :server:test
  - After validation, commit your real tests. For this template task, we created and ran such a test and removed it afterward; see 4. Audit trail.

### Ktor application test pattern (existing, verified)
- ApplicationTest demonstrates testing an in-memory Ktor app:
  - testApplication { application { module() } ... }
  - client.get("/") then assert status and body.
- Use this pattern to cover new routes/middleware without binding to a real port.

### Compose/KMP tests
- There is a placeholder test source set at composeApp/src/commonTest. If you add common tests, prefer pure business-logic tests without relying on platform-specific UI/runtime unless you configure appropriate runners for Android/iOS.

## Additional development information

### Static analysis (Detekt)
- Configured once at root for all modules. Run manually with: ./gradlew detekt
- Configuration file: tools/detekt/detekt-config.yml
- Sources include KMP source sets per root build.gradle.kts getDetektSourcePaths(). If you add a new source set path, ensure it exists so detekt doesn’t scan non-existent dirs.

### BuildKonfig and build-time metadata
- composeApp uses BuildKonfig to expose build metadata (app name, version, IS_DEBUGGABLE, EFFECTIVE_BUILD_TYPE). Values originate from application.properties and environment-derived build type logic (see 1.3). When adding features that depend on build type, prefer reading from BuildKonfig rather than ad-hoc environment checks.

### Logging and Error Reporting skeleton
- composeApp contains a logging/error reporting abstraction (composeApp/.../logging/StashConfiguration.kt):
  - In debug builds (Platform.isDebuggable=true), LogErrorReporting is used (currently stubbed with commented Log calls).
  - In non-debug, SentryBootstrapper.initialize is invoked with SentryConfiguration, and a SentryErrorReporting instance is wrapped with toggling and breadcrumb decorators.
  - Actual Sentry SDK calls are commented out. To activate Sentry:
    - Add Sentry dependencies to relevant targets.
    - Provide SentryConfiguration with a valid DSN and desired options.
    - Un-comment and wire the Sentry calls in SentryErrorReporting and SentryBootstrapper.
  - You can also switch to ErrorReportingImplementation.NoOp for local experimentation.

### Android flavors/build types
- debug, release, and a staging buildType exist. Staging inherits from release and uses the debug signingConfig for convenience.
- Bundle splits for language are disabled to keep all languages in bundles.
- Lint: warnings are treated as errors; abortOnError=false to avoid failing builds for warnings.

### Wasm/JS dev ergonomics
- The wasm dev server serves project sources for debugging (composeApp/build.gradle.kts config adds rootDir and projectDir to static devServer paths). Source maps work in-browser for easier stepping through Kotlin/Compose code.

### CI hints
- The project badge references GitHub Actions (build-all.yml). If adding steps, keep tasks granular (e.g., :server:test for quick validation, platform builds behind a separate job/matrix).

## Common pitfalls
- Missing google-services.json will break Android builds; ensure it’s present (it is in this repo).
- Release signing requires local.properties; debug/staging do not.
- Compose/Wasm tasks may try to fetch Node/webpack tooling; rely on Gradle-managed toolchain (no manual Node setup usually required).
- If detekt starts failing after adding new source sets, verify getDetektSourcePaths includes your directories and they exist on disk (the helper filters non-existent paths).

# Project Guidelines

These guidelines capture project-specific knowledge to help advanced contributors work efficiently on Now in Android within this repository.

## Build and Configuration
- Tooling requirements:
    - JDK 17+ is required (enforced in settings.gradle.kts).
    - Gradle 9.x wrapper is included; invoke via `./gradlew`.
    - Android Studio (latest stable) recommended for IDE tasks; select the `app` run configuration and use `demoDebug` for local development.
- Variants and flavors:
    - Build types: `debug`, `release` (and `benchmark` for baseline profile generation).
    - Product flavors: `demo` (offline static content) and `prod` (requires backend not publicly available). Prefer `demoDebug` locally.
- Typical build commands:
    - Assemble demo debug: `./gradlew :app:assembleDemoDebug`
    - Assemble all: `./gradlew assembleDemoDebug` (multi-module as needed)
    - Release build (compose metrics optional):
        - `./gradlew assembleRelease -PenableComposeCompilerMetrics=true -PenableComposeCompilerReports=true`
        - Reports: `build/compose-reports`, Metrics: `build/compose-metrics` (paths relative to each module).
- Code formatting and linting (Spotless via init script):
    - Apply formatting: `./gradlew --init-script gradle/init.gradle.kts spotlessApply`
    - The init script configures ktlint and license headers across Kotlin, KTS, and XML.
- Compose compiler configuration:
    - See `compose_compiler_config.conf` for project-level Compose compiler knobs; metrics/reports can be enabled via the Gradle properties above.

## Testing
The project contains a mixture of JVM unit tests, instrumentation tests (Gradle Managed Devices), and screenshot tests (Roborazzi). Important notes:
- Only `demoDebug` is generally supported for end-to-end and screenshot testing locally (per README). Avoid running test tasks across all variants.
- Hilt is used for DI; many tests rely on test doubles rather than mocking libraries.

### How to run tests
- Run all JVM unit tests for the common, supported variant:
    - `./gradlew testDemoDebug` (runs unit tests for modules with that variant)
- Run JVM unit tests for a single module (example for a pure JVM module):
    - `./gradlew :core:common:test`
- Run instrumentation tests with Gradle Managed Devices (GMD):
    - Typical task names are generated; examples used in this repo:
        - `./gradlew pixel6api31aospDebugAndroidTest`
        - Other options (depending on configured devices): `pixel4api30aospatdDebugAndroidTest`, `pixelcapi30aospatdDebugAndroidTest`
    - These target debug builds; prefer `demoDebug` flavor when applicable for end-to-end.
- Screenshot tests (Roborazzi):
    - Verify screenshots: `./gradlew verifyRoborazziDemoDebug`
    - Record new screenshots: `./gradlew recordRoborazziDemoDebug`
    - Compare differences: `./gradlew compareRoborazziDemoDebug`
    - Tip: On non-Linux platforms, recorded images may differ subtly. Record on `main` before changes to reduce noise.

### Creating and running a new unit test (demonstrated and verified)

2. Run the module’s unit tests:
    - `./gradlew :core:common:test`

    - Gradle builds and the test suite passes (BUILD SUCCESSFUL). This was executed and confirmed during guideline preparation.

    - Remove the temporary test file after verification (left no extra files in the repository as per task requirements).

### Adding more tests
- JVM tests
    - Prefer placing simple logic and pure Kotlin utilities in JVM-only modules (e.g., `:core:common`) to keep tests fast and device-independent.
    - For coroutine-based flows, use `kotlinx.coroutines.test` and `cashapp/turbine` (already configured in `:core:common`).
- Android instrumented tests
    - Place under `src/androidTest` of the Android module; tasks are auto-disabled for modules without `androidTest` (see `AndroidInstrumentedTests.kt`).
    - For feature/UI tests, stick to `ComposeTestRule` with `ComponentActivity`. Larger app-level tests can target `:app` and start `MainActivity`.
- Screenshot tests
    - Live under `app/src/testDemo/kotlin/...` with Roborazzi. Ensure you record new baselines after intentional UI changes.

# Now in Android Project

Now in Android is a native Android mobile application written in Kotlin. It provides regular news
about Android development. Users can choose to follow topics, be notified when new content is
available, and bookmark items.


## Modules

The main Android app lives in the `app/` folder. Feature modules live in `feature/` and core and shared modules in `core/`.

## Commands to Build & Test

The app and Android libraries have two product flavors: `demo` and `prod`, and two build types: `debug` and `release`.

- Build: `./gradlew assemble{BuildType}`. Typically `assembleDebug`.
- Fix linting/formatting: `./gradlew --init-script gradle/init.gradle.kts spotlessApply`
- Run local tests: `./gradlew {BuildType}Test`
- Run single test: `./gradlew {BuildType}Test --tests "com.example.myapp.MyTestClass"`
- Run local screenshot tests: `./gradlew verifyRoborazziDemoDebug`

### Instrumented tests

- Gradle-managed devices to run on device tests: `./gradlew pixel6api31aospDebugAndroidTest`. Also `pixel4api30aospatdDebugAndroidTest` and `pixelcapi30aospatdDebugAndroidTest`.

### Creating tests

#### Instrumented tests

- Tests for UI features should only use `ComposeTestRule` with a `ComponentActivity`.
- Bigger tests live in the `:app` module, and they can start activities like `MainActivity`.

#### Local tests

- [kotlinx.coroutines](https://github.com/Kotlin/kotlinx.coroutines) for most assertions
- [cashapp/turbine](https://github.com/cashapp/turbine) for complex coroutine tests
- [google/truth](https://github.com/google/truth) for assertions

## Continuous integration

- The workflows are defined in `.github/workflows/*.yaml` and they contain various checks.
- Screenshot tests are generated by CI, so they shouldn't be checked into the repo from a workstation.

## Version control and code location

- The project uses git and is hosted in https://github.com/android/nowinandroid.
